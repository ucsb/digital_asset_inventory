# Scanner Workflow Specification

## Overview

The Digital Asset Scanner (`DigitalAssetScanner` service) discovers and catalogs all digital assets across a Drupal site. The scanning process runs in seven sequential phases, each targeting a different asset source or post-processing step. The scanner uses Drupal's Batch API with time-budgeted processing to handle large datasets without timeouts.

## Scan Initiation

**Route**: `/admin/digital-asset-inventory/scan`
**Form**: `ScanAssetsForm`
**Permission**: `scan digital assets`

### Pre-Scan Steps

1. Acquire scan lock (concurrency protection)
2. Reset scan statistics (`resetScanStats()`)
3. Store scan start timestamp in Drupal State
4. Suspend automated cron (`suspendCron()`) to prevent cron interference during batch processing
5. Initialize batch operations for all seven phases

## Scan Phases

### Phase 1: Managed Files

**Method**: `scanManagedFilesChunk($context, $is_temp)`
**Source**: `file_managed` database table
**Processing**: Time-budgeted with ID-based cursor (`last_fid`)

#### Process

```text
FOR EACH file in file_managed (cursor-based, time-budgeted):
  1. Skip system-generated files (see Exclusions below)
  2. Determine asset_type from MIME type
  3. Determine category from asset_type
  4. Check for Media entity association (file_usage table)
  5. Convert URI to absolute URL
  6. Detect private file system (private://)
  7. Create/update digital_asset_item (is_temp=TRUE)
  8. Detect and record usage:
     - Entity reference fields targeting media (bulk via getMediaReferenceFieldTables cache)
     - CKEditor media embeds (<drupal-media> tags)
     - Text field file links (href/src via getTextFieldTables cache)
     - Direct file/image field usage (via getFileFieldTables cache)
     - HTML5 <video>/<audio> tag detection
  9. Buffer usage records for bulk INSERT (flushed at callback exit)
```

**Performance features:**
- Paragraph parent cache eliminates repeated parent lookups within a callback
- Usage records buffered in memory, bulk-flushed via raw SQL at callback exit
- Orphan reference records similarly buffered and bulk-flushed
- Field table lists cached per PHP process (text fields, file fields, media ref fields)

#### File Exclusions

System-generated files are excluded from scanning:

| Pattern | Description | Source | Rationale |
| ------- | ----------- | ------ | --------- |
| `styles/` | Image style derivatives | Drupal core | Regenerated by Drupal on demand |
| `thumbnails/` | Media thumbnail images | Contrib/custom | Not a Drupal core default. Used by contrib modules (e.g., `media_pdf_thumbnail`) and custom `thumbnails_directory` configurations for oEmbed media sources. Common enough in the ecosystem to exclude by convention |
| `media-icons/` | Media type placeholder icons | Drupal core | Shipped with Drupal core Media module |
| `oembed_thumbnails/` | oEmbed preview images | Drupal core | Default `thumbnails_directory` for oEmbed media sources. Auto-fetched from remote providers |
| `video_thumbnails/` | Video preview images | Contrib | Created by `video_embed_field` contrib module. Not a Drupal core directory |
| `css/` | Aggregated CSS | Drupal core | Regenerated on cache rebuild |
| `js/` | Aggregated JavaScript | Drupal core | Regenerated on cache rebuild |
| `php/` | PHP temporary files | Drupal core | Transient runtime files |
| `ctools/` | CTools generated files | Contrib | Created by CTools contrib module. Not a Drupal core directory |
| `xmlsitemap/` | Sitemap files | Contrib | Created by XML Sitemap contrib module. Not a Drupal core directory |
| `config_*` | Configuration exports | Drupal core | Config sync directories |
| `wordmark/` | Site logos | Site-specific | Not a Drupal core directory. Site-specific branding directory used by the active theme. Sites not using this convention should remove this exclusion or replace it with their own theme-specific branding directory |

All excluded directories contain auto-generated files that are either regenerated on demand, shipped with Drupal, or auto-fetched from external providers. User-uploaded content directories are never excluded.

**Note:** Directories marked **Contrib** or **Site-specific** in the Source column are not created by Drupal core. They are included in the exclusion list because they are common in the Drupal ecosystem and always contain auto-generated content, never user-uploaded files. Sites that do not use these contrib modules can safely leave the exclusions in place (they will simply match nothing).

**Note:** The `thumbnails/` exclusion does not prevent thumbnail usage detection. Media entity thumbnails are discovered through entity relationships in Phase 1 (forward detection via `file_usage` and reverse detection via media entity query), not through filesystem scanning. See the [Derived Media Thumbnail Usage Detection Spec](derived-media-thumbnail-usage-detection-spec.md) for details.

#### Custom Excluded Directories

In addition to the built-in list, site administrators can add custom directory exclusions via **Settings вҶ’ Scanner Settings вҶ’ Additional excluded directories**. Enter one directory name per line (e.g., `pdf-previews`). Each entry is excluded from both `public://` and `private://` file systems and applies to both the filesystem orphan scan and the managed file query.

The built-in exclusion list is not editable вҖ” it is always applied. Custom exclusions are merged with the built-in list at scan time.

**Configuration key:** `scan_excluded_directories` (sequence of directory names)

**Input format:**
- One top-level directory name per line
- Nested paths are not supported
- Stream wrapper prefixes, slashes, and wildcards are automatically stripped
- Single-character names are rejected (too broad)

**Validation:** On save, the form checks if any active or queued archive records reference files in the directories being excluded. If so, a warning is displayed (non-blocking): the archive records themselves are not affected, but the assets will no longer appear in the inventory after the next scan.

**Archive interaction:** Archive records (`digital_asset_archive`) are self-contained entities that store their own `original_fid`, `original_path`, and `file_checksum`. They do not depend on `digital_asset_item` records. If an excluded directory contains archived files:
- The archive records remain intact and functional
- `validateArchivedFiles()` continues to detect missing/modified files
- The assets will not appear in inventory views (no `digital_asset_item` is created during scan)

### Phase 2: Orphan Usage Index

**Method**: `buildOrphanUsageIndex($context)`
**Source**: All text-field tables (LIKE scan for `/files/`)
**Processing**: Time-budgeted, one table per iteration

#### Purpose

Pre-builds a usage map for orphan files before Phase 3 processes them. This replaces the previous approach of running per-file LIKE queries (O(files Г— tables)) with a single pass over all text-field tables (O(tables)), dramatically reducing database load on large sites.

#### Process

```text
1. Build orphan file list from filesystem (files not in file_managed)
2. Build search needles from orphan file URIs
3. FOR EACH text-field table:
   a. SELECT rows WHERE value_column LIKE '%/files/%'
   b. Match each row's text against orphan file needles
   c. Record (entity_type, entity_id, field_name) вҶ’ url_hash mapping
4. Store completed usage map in State API for Phase 3
5. Store orphan file list in State API for Phase 3
```

**Performance note:** On a site with 2,800+ files and hundreds of text-field tables, this phase typically completes in under 30 seconds. The usage map enables Phase 3 to attach usage records without any additional database queries per orphan file.

### Phase 3: Orphan Files

**Method**: `scanOrphanFilesChunkNew($context, $is_temp)`
**Source**: Filesystem (`public://`, `private://`) + Phase 2 usage index
**Processing**: Time-budgeted with index-based cursor, two-pass bulk writes

#### Purpose

Discovers files uploaded outside Drupal (FTP, SFTP, direct upload) that are not tracked in `file_managed`.

#### Process

```text
FOR EACH file on filesystem (from Phase 2's pre-built list):
  1. Skip excluded directories (same as Phase 1)
  2. Match file extension against known types
  3. Determine MIME type from extension
  4. Determine asset_type and category
  5. Create digital_asset_item with source_type='filesystem_only'
  6. Look up usage from Phase 2's pre-built usage map (no DB queries)
  7. Buffer items and usage records for bulk INSERT
  8. Flush buffers at sub-batch boundaries
```

#### Supported Extensions

Documents: pdf, doc, docx, xls, xlsx, ppt, pptx, txt, csv
Images: jpg, jpeg, png, gif, svg, webp
Videos: mp4, webm, mov, avi
Audio: mp3, wav, m4a, ogg
Compressed: zip, tar, gz, 7z, rar

### Phase 4: External URLs

**Method**: `scanContentChunk($context, $is_temp)`
**Source**: Text and link field tables
**Processing**: Time-budgeted with compound cursor (table_index + last_entity_id)

#### Purpose

Discovers external resources (Google Docs, YouTube, etc.) referenced in content.

#### Scanned Entity Types

| Prefix | Entity Type | Examples |
| ------ | ----------- | -------- |
| `node__` | Nodes | Articles, Pages, Events |
| `paragraph__` | Paragraphs | Text blocks, accordions, carousels |
| `taxonomy_term__` | Taxonomy terms | Categories, tags |
| `block_content__` | Custom blocks | Sidebar content, footer blocks |

**Not scanned**: User profiles, comments, system configuration entities, or custom entity types not listed above.

#### Process

```text
FOR EACH of the four entity types above:
  1. Find all field tables with that prefix
  2. FOR text fields (_value columns):
     - Extract URLs using regex pattern matching
  3. FOR link fields (_uri columns):
     - Read URL directly from field value
  4. FOR EACH extracted URL:
     a. Match URL against configured patterns
     b. IF matches known service (not 'other'):
        - Create/update digital_asset_item with source_type='external'
        - Record usage on parent entity
```

#### URL Pattern Matching

URLs are matched against service-specific patterns:

| Service | Pattern Examples |
| ------- | ---------------- |
| Google Docs | docs.google.com/document |
| Google Sheets | docs.google.com/spreadsheets |
| Google Drive | drive.google.com |
| YouTube | youtube.com, youtu.be |
| Vimeo | vimeo.com |
| Box | box.com |
| Dropbox | dropbox.com |
| Qualtrics | qualtrics.com |
| Canvas | instructure.com |

**Note**: URLs that don't match any known pattern are skipped (not inventoried as 'other').

### Phase 5: Remote Media

**Method**: `scanRemoteMediaChunk($context, $is_temp)`
**Source**: Media entities with oEmbed source
**Processing**: Time-budgeted with ID-based cursor (`last_mid`)

#### Purpose

Discovers remote video media (YouTube, Vimeo) added via Media Library's "Remote Video" media type.

#### Why Separate Phase?

Remote video media entities:
- Don't have entries in `file_managed` (no physical file)
- Store video URL in source field (not file reference)
- Need dedicated detection logic

#### Process

```text
1. Identify media types using oEmbed source plugin (oembed:video, video_embed_field)
2. FOR EACH remote media entity:
   a. Extract video URL from source field
   b. Detect asset type from URL (youtube, vimeo)
   c. Create digital_asset_item:
      - source_type = 'media_managed'
      - filesize = NULL (no local file)
      - file_name = media title
   d. Find usage via:
      - Entity reference fields
      - CKEditor media embeds
   e. Buffer usage records for bulk INSERT
```

### Phase 6: Menu Links

**Method**: `scanMenuLinksChunk($context, $is_temp)`
**Source**: `menu_link_content` entities
**Processing**: Time-budgeted with ID-based cursor (`last_id`)

#### Purpose

Discovers file references in navigation menus. Menu links can contain direct links to files and documents (PDFs, policies, forms) that are accessible without going through node content.

#### Why Separate Phase?

Menu link content entities:
- Are not covered by the standard content entity scan
- Can link directly to files using internal URIs, base URIs, or full URLs
- Represent user-accessible download paths that must be tracked for compliance

#### Process

```text
1. Query all menu_link_content entities
2. FOR EACH menu link:
   a. Extract URI from link field
   b. Normalize URI format (construction uses site-aware path via FilePathResolver):
      - internal:{public_files_path}/...
      - base:{public_files_path_no_slash}/...
      - https://example.com{public_files_path}/...
   c. IF URI points to a file:
      - Match against known asset in inventory
      - Create usage record linking menu link to asset
      - Store menu name for display context
   d. Handle both public and private file paths (/system/files/...)
```

#### Supported URI Formats

| Format | Example (default path shown; actual path is site-aware via FilePathResolver) |
| ------ | ------- |
| Internal URI | `internal:/sites/default/files/doc.pdf` |
| Base URI | `base:sites/default/files/doc.pdf` |
| Full URL (local) | `https://example.com/sites/default/files/doc.pdf` |
| Multisite URI | `internal:/sites/mysite.com/files/doc.pdf` |
| Private file | `internal:/system/files/private/doc.pdf` |

### Phase 7: CSV Export Fields

**Method**: `updateCsvExportFieldsBulk($context)`
**Source**: `digital_asset_item` (temp rows) + `digital_asset_usage`
**Processing**: Time-budgeted with ID-based cursor (`csv_last_id`)

#### Purpose

Populates denormalized CSV export columns (`filesize_formatted`, `active_use_csv`, `used_in_csv`) on all temporary asset items. This is deferred to a separate final phase so that all usage records from Phases 1вҖ“6 are complete before generating the export strings.

#### Why Separate Phase?

Previously, CSV export fields were computed inline during Phase 1. This required loading parent entities for label resolution during the main scan, adding significant overhead and entity cache pressure. Deferring to Phase 7:
- Eliminates redundant entity loads during scanning phases
- Uses bulk `loadMultiple()` for efficient parent entity resolution
- Ensures `used_in_csv` reflects complete usage from all prior phases
- Runs after all usage is finalized, producing accurate "Active Use Detected" values

#### Process

```text
FOR EACH digital_asset_item WHERE is_temp=TRUE (cursor-based):
  1. Bulk-load usage records for a batch of 50 items
  2. Bulk-load parent entities by type for label resolution
  3. Format filesize (bytes вҶ’ "2.5 MB")
  4. Compute active_use_csv ("Yes"/"No")
  5. Build used_in_csv string ("Page Title (URL), ...")
  6. Bulk-UPDATE all 50 items via raw SQL
```

## Post-Scan Processing

### On Success (`batchFinished` with $success=TRUE)

1. **Write legacy orphan count** for backward compatibility
2. **Promote temporary items** (`promoteTemporaryItems()`)
   - Delete old permanent items and their usage records
   - Mark new temp items as permanent
3. **Clear checkpoint state**
4. **Store completion timestamp**
5. **Calculate scan duration**
6. **Query actual counts** by source type
7. **Restore automated cron** (`restoreCron()`)
8. **Release scan lock**
9. **Display summary message**:
   - Total assets found
   - Breakdown by source type
   - Usage record count
   - Orphaned paragraph count (if any)

### On Failure/Cancel (`batchFinished` with $success=FALSE)

1. **Preserve temporary items** (for potential resume)
2. **Preserve checkpoint** (for potential resume)
3. **Restore automated cron** (`restoreCron()`)
4. **Release scan lock**
5. **Log warning**
6. **Display message**: "Scan interrupted. You can resume from where it left off on the scan page."

## Performance Features

### Time-Budgeted Processing

Each batch callback runs within a configurable time budget (default: 10 seconds). When the budget is reached, the callback yields control back to the Batch API. This prevents HTTP timeouts on hosting platforms with strict request limits (e.g., Pantheon's ~59-second timeout).

**Configuration**: Settings вҶ’ Scanner Settings вҶ’ Batch time budget (1вҖ“30 seconds)

| Environment | Recommended Budget | Rationale |
|-------------|-------------------|-----------|
| Pantheon | 4вҖ“8s | ~59s request timeout |
| Acquia | 4вҖ“8s | Similar timeout to Pantheon |
| Platform.sh | 4вҖ“8s | 60s default timeout |
| Dedicated VPS / AWS | 10вҖ“15s | Higher timeouts, fewer round-trips |
| Local / DDEV / Lando | 15вҖ“20s | No practical timeout |

### Heartbeat-Based Monitoring

The scanner updates a heartbeat timestamp every 2 seconds during active processing. This prevents false stale-lock detection during long operations and provides visibility into scan progress.

### Cron Suspension

Automated cron is suspended during scanning to prevent cron jobs from competing for database resources. The original cron interval is saved in State and restored when the scan completes (even after interruption and resume).

### Bulk Operations

- **Bulk writes**: Usage records and orphan references are buffered in memory and flushed via raw SQL INSERT at callback exit
- **Bulk reads**: Entity references resolved via `loadMultiple()` instead of per-entity loads
- **Cached field tables**: Text field, file field, and media reference field table lists are cached per PHP process
- **Paragraph parent cache**: Parent entity lookups cached per callback to avoid redundant tracing

### Memory Management

Entity storage caches are reset at the end of each batch callback. `drupal_static_reset()` is called when 50+ items are processed in a callback to prevent unbounded memory growth during large scans.
